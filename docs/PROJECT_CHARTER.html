§ 2

The One-Line Thesis

Encode professional project patterns as executable, governed workflows: agent proposes, deterministic tools dispose, human governs, every claim traceable.

The Delivery Engine takes a dataset and a plain-English goal and produces a complete, professional project package — notebook, report, presentation, workpaper, README — where every artifact meets an evidence standard, not just a plausibility standard.

🤝

Built by directing Claude as AI collaborator

Mohd Saif Hussain — a data annotation and operations specialist — directed Claude across 23 documented build steps. The AI wrote code. The human governed every decision: setting the scope, approving each step, declining wrong runs before they became evidence. Every architectural and security decision is human-made and source-verified. The build record is public, timestamped, and re-performable in this charter.

§ 4

Architecture Principles

These are not preferences. Each is stated with its rationale. Violating any of them means the build has failed regardless of how well it demos.

§ 4.1 — The Injected-Numbers Rule

AI never writes a number

The AI never computes, estimates, or writes a number. Every figure in every generated artifact is injected from the Findings Store and carries its hash. AI writes prose, structure, and code — never facts.

A report where the AI wrote "churn is 23.4%" and nobody verified it is worse than no report, because it looks professional. This rule must be architecturally enforced, not enforced by prompt instruction.

§ 4.2 — Deterministic Quality Gates

A failed gate stops everything

AnalystKit and OpsKit run as must-pass gates. A failed gate stops the pipeline. No AI stage may override, soften, or route around a failed gate.

A control that cannot fail is not a control.

§ 4.3 — The Findings Store is the Audit Boundary

Same findings → same hash → re-performable evidence

All deterministic outputs land in canonical JSON (sort_keys), SHA-256 hashed. These are the only numbers any AI stage may consume.

Pattern proven at small scale in AnalystKit v2.0. This project scales it up; it does not reinvent it.

§ 4.4 — Two Human Gates, Architecturally Placed

Human judgment cannot be bypassed

Gate 1: The Planner's selected playbook and execution plan are shown for approval before anything executes.
Gate 2: Any AI-authored content that feeds back into the deterministic layer requires explicit approval by SHA-256 before use.

Gate 2 exists because AI-authored rules feeding the deterministic layer is the one place the audit boundary could be poisoned. The highest-risk architectural point.

§ 4.5 — Playbooks, Not Code

Adding an archetype never modifies the engine

Project archetypes are declared as TOML playbooks. Adding a new project type means writing a new playbook. The engine stays small, tested, gated; the pattern library grows.

Users can encode their own professional patterns as playbooks. The product is not just our patterns — it is a format for professionals to encode theirs.

§ 4.6 — The Planner is 80% Deterministic

LLM resolves only genuine ambiguity

Project-type classification is decided by rules over the AnalystKit profile output. The LLM resolves only genuine ambiguity in goal wording, and its decision plus rationale is logged.

"Decide the project type" is a deterministic problem wearing an AI costume.

§ 4.8 — The Success Test

A reviewer who was never in the room can re-perform any stage

A reviewer who was never in the room can re-perform any stage from the delivery package alone and get the same hashes.

If they can, it is evidence. If they cannot, it is just output.

§ 4.4

Human Gates

🔒
Human Gate 1 · Plan Approval

Before anything executes

The Planner's selected playbook and execution plan are shown to the human for approval. Declining stops the run with an audit entry before any stage runs. The rendered preview is written to execution_preview.md and hashed — what the human was shown is itself evidence.

🔐
Human Gate 2 · Content-Bound Approval

Before AI content enters the deterministic layer

Engine-drafted validation rules await approval before feeding back into the deterministic pipeline. Approval quotes the draft's exact SHA-256 — you cannot approve a version you have not seen. The examples auto-approve for demo purposes. Real use requires a human to open rules_draft.json, read the proposed rules, verify they match the domain, and re-run with the approved hash.

v0.14 Step 18 — Analyst-Error Guardrails G1–G6 272 ✓
Six controls, each traceable to published research on how analysts actually fail, several motivated by failures observed first-hand in this engine's own production runs.

G1 Leakage Sentinel — per-feature association with target; flags possible leakage, NEVER gates.

G3 Minimum Detectable Effect — "not significant" can never silently mean "no effect."
G4 Analyst-Bias Checklist — selection/survivorship, denominators, mixed granularity.
G5 Source Fingerprint — SHA-256 of the INPUT dataset before any stage runs.
G6 Limitations & Assumptions — assembled from caveats in hashed findings, NEVER fabricated. 272 tests.
